# MKM 商业化与增长路径

## 1. 文档目标

本文档定义 MKM 从学习项目走向真实产品后的商业化和增长思路。

注意：V1 的核心目标仍然是学习和验证，不建议过早商业化。

商业化应遵循：

```text
先验证真实使用价值，再设计收费模式。
```

---

## 2. 产品商业定位

MKM 的潜在商业定位：

> 面向程序员和技术学习者的移动端优先 Markdown 知识库 + AI 助手。

核心差异点：

| 差异点 | 说明 |
|---|---|
| Android 优先 | 手机端 Markdown 阅读和轻整理体验更好 |
| 程序员导向 | 强化代码块、技术文档、README、CLI、AI Agent |
| AI 知识整理 | AI 总结、生成 Markdown、提取待办 |
| BYOK 模式 | 用户自带 API Key，降低平台成本 |
| 公开知识库 | 未来可做技术内容分享和社区 |

---

# 4. 商业化原则

## 4.1 V1 不收费

V1 阶段建议免费。

原因：

- 产品还在验证期
- 功能稳定性还需要打磨
- 用户规模小
- 收费会增加使用门槛

---

## 4.2 V2 可以引入高级功能

当用户开始持续使用后，再考虑高级功能。

可以收费的不是“基础 Markdown 查看”，而是：

- 云同步容量
- AI 高级能力
- 公开知识库高级展示
- 多端同步
- 数据导出/备份增强
- 团队空间（后期）

---

# 5. 可选商业模式

## 5.1 Freemium 免费增值

基础功能免费，高级功能收费。

### 免费版

- 本地 Markdown 查看
- 基础文档管理
- 少量云同步空间
- 用户自带 API Key 的 AI 功能
- 基础待办

### Pro 版

- 更大云同步空间
- 更多公开知识库数量
- AI 对话历史保存
- 高级导出
- 高级主题
- 高级搜索

适合 MKM。

---

## 5.2 BYOK + 云同步收费

AI 模型费用由用户自己承担，MKM 收取云服务和同步能力费用。

优点：

- 平台不用承担 AI 成本
- 用户对成本可控
- 适合个人开发者运营

收费点：

```text
云同步空间
公开知识库数量
多设备同步
自动备份
```

---

## 5.3 一次性买断

例如：

```text
¥29 / ¥49 一次性解锁 Pro
```

适合早期小工具，但长期维护收入有限。

---

## 5.4 订阅制

例如：

```text
¥9/月
¥59/年
```

适合产品成熟后。

订阅权益可以包括：

- 云同步空间
- 高级 AI 流程
- 知识库公开页
- 自动备份
- 多端同步增强

---

# 6. 推荐商业化路线

## 阶段 1：免费内测

时间：V1

目标：验证用户是否真的愿意用。

```text

关注留存和真实使用场景
```

时间：V1.5 / V2

目标：扩大用户，验证核心差异点。

```text

```

Pro 权益：

- 更大云同步容量
- 更多公开知识库
- AI 对话历史保存
- Markdown 高级导出
- 自动备份
- 高级主题

---

## 阶段 4：团队版（远期）

- 评论协作

# 7. 增长路径

## 7.1 第一阶段：技术圈内测

渠道：

- 技术朋友
- 微信群 / QQ 群
- GitHub README
- 掘金 / 知乎 / CSDN 开发日志
- 酷安发布测试版

内容形式：

```text
我做了一个 Android 端 Markdown 知识库工具
可以用 AI 帮你总结技术文档、生成待办
```

---

## 7.2 第二阶段：开源吸引用户

可以考虑开源部分代码：

- Android 客户端
- Web 客户端
- 后端核心 API

开源优势：

- 程序员信任度更高
- 容易获得反馈
- 适合学习项目展示
- 有利于吸引贡献者

需要注意：

- 不要提交密钥
- 不要泄露用户数据
- 商业化前确认开源协议

推荐协议：

```text
Apache-2.0 或 MIT
```

---

## 7.3 第三阶段：内容增长

围绕 MKM 可以产出内容：

- Android Markdown 渲染实践
- Spring Boot + Vue + Android 全栈项目实战
- 如何接入 DeepSeek 到个人知识库
- AI 如何生成 Markdown 学习笔记
- 从 Obsidian 到自研知识库的学习路径

这些内容可以带来自然用户。

---

## 7.4 第四阶段：公开知识库生态

当 V3 做公开知识库后，增长可以来自用户生成内容。

例如：

```text
公开我的 Android 学习知识库
公开 Spring Boot 学习路线
公开 AI Agent 开发笔记
```

用户分享公开知识库，本身就是传播入口。

# 9. 产品风险

## 9.1 功能过多导致烂尾

风险：

```text
Markdown、AI、待办、知识库、Web、后端、CLI 全做，容易失控。
```

应对：

```text
先跑通 Markdown + AI + 待办三条闭环，其他功能只做入口。
```

---

## 9.2 与 Obsidian 差异不够

风险：

```text
用户会问：为什么不用 Obsidian？
```

应对：

强调差异：

- Android 移动端优先
- 程序员友好
- BYOK AI 助手
- CLI / Agent 调用
- 公开技术知识库

---

## 9.3 AI 成本不可控

风险：

```text
如果平台代付 AI 费用，成本会上升。
```

应对：

```text
V1/V2 坚持 BYOK，用户自己配置 API Key。
```

---

## 9.4 用户数据安全

风险：

```text
文档和 API Key 都是敏感信息。
```

应对：

- 默认私有
- API Key 加密
- 支持导出
- 支持删除账号
- 隐私政策清晰说明

