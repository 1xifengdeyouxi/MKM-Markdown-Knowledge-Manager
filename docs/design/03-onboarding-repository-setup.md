# MKM 首次启动与仓库初始化 UI 设计

## 1. 文档目标

本文档定义 MKM Android 端首次启动时的 Onboarding 流程。

新的产品原则：

```text
App 可以不登录使用。
账号是可选的。
云同步需要登录/注册账号。
首次启动的第一任务不是登录，而是选择数据存储位置和仓库方式。
```

因此，V1 的启动入口应从原来的 `LoginActivity` 调整为：

```text
OnboardingActivity / RepositorySetupActivity
```

登录注册不再是强制第一屏，而是在用户选择“开启同步”或“从账号恢复仓库”时出现。

---

# 2. 核心概念

## 2.1 仓库 Repository

MKM 中的“仓库”类似 Obsidian 的 Vault。

一个仓库代表一组 Markdown 文档、待办、AI 配置和本地索引数据。

```text
Repository
├── Markdown 文档
├── Todo 数据
├── AI 生成结果
├── 本地配置
└── 同步状态
```

V1 可以先支持单仓库，后续再扩展多仓库切换。

---

## 2.2 本地优先原则

MKM 应支持不登录直接使用：

```text
首次启动 → 创建本地仓库 → 进入 App → 本地 Markdown 管理
```

这样可以降低使用门槛，也更接近 Obsidian 的体验。

---

## 2.3 同步原则

云同步是增强能力，不是启动前置条件。

```text
本地仓库：无需账号
云同步：必须登录/注册账号
```

当用户选择开启同步时，再进入账号流程。

---

# 3. 首次启动总流程

```text
App 启动
    │
    ├─ 已完成仓库初始化
    │       └─ 进入 MainActivity
    │
    └─ 未完成仓库初始化
            └─ 进入 OnboardingActivity
                    │
                    ├─ 选择语言
                    │
                    ├─ 选择仓库方式
                    │       ├─ 创建新仓库
                    │       │       └─ 选择是否开启同步
                    │       │               ├─ 开启同步 → 登录/注册 → 设置仓库信息
                    │       │               └─ 跳过同步 → 设置仓库信息
                    │       │
                    │       └─ 使用现有仓库
                    │               ├─ 选择本地已有仓库目录
                    │               └─ 登录账号，从云端恢复仓库
                    │
                    └─ 完成 → 进入 MainActivity
```

## 6.1 页面目标

“使用现有仓库”同时支持两种场景：

1. 选择设备上的已有本地仓库目录
2. 登录账号，从云端同步恢复已有仓库

---

## 6.2 页面结构

```text
┌────────────────────────────┐
│ ← 使用现有仓库              │
├────────────────────────────┤
│ 选择仓库来源                │
│                            │
│ ┌────────────────────────┐ │
│ │ 📁 打开本地仓库          │ │
│ │ 选择设备上已有的 MKM 仓库 │ │
│ └────────────────────────┘ │
│                            │
│ ┌────────────────────────┐ │
│ │ ☁ 从账号恢复             │ │
│ │ 登录后从云端同步仓库      │ │
│ └────────────────────────┘ │
│                            │
└────────────────────────────┘
```

---

## 6.3 打开本地仓库

点击“打开本地仓库”后：

```text
调用 Android 系统目录选择器
    │
    ├─ 用户选择目录
    │       ├─ 校验是否为 MKM 仓库
    │       │       ├─ 是 → 导入仓库 → 进入 MainActivity
    │       │       └─ 否 → 提示“该目录不是有效的 MKM 仓库”
    │       │
    │       └─ 保存目录访问权限
    │
    └─ 用户取消 → 返回来源选择页
```

Android 技术建议：

```text
使用 Storage Access Framework：ACTION_OPEN_DOCUMENT_TREE
保存 persistableUriPermission
```

避免直接依赖传统外部存储路径权限。

---

## 6.4 从账号恢复

点击“从账号恢复”后：

```text
进入登录/注册页
    │
    └─ 登录成功
            │
            ├─ 请求云端仓库列表
            │       ├─ 有仓库 → 选择要恢复的仓库
            │       └─ 无仓库 → 引导创建新仓库
            │
            └─ 选择仓库 → 同步到本地 → 进入 MainActivity
```

# 9. 仓库位置设计

## 9.1 设备存储

说明：

```text
仓库保存在用户选择的设备目录中，其他应用可以访问。
```

适合用户：

- 希望用文件管理器看到 Markdown 文件
- 希望用其他编辑器编辑 Markdown
- 希望通过电脑、Syncthing、坚果云等工具同步目录
- 习惯 Obsidian Vault 模式

UI 文案：

```text
设备存储
保存在你选择的文件夹中，其他应用可以访问。
适合需要用文件管理器或其他 Markdown 工具打开的用户。
```

技术建议：

```text
使用 ACTION_OPEN_DOCUMENT_TREE 选择目录
通过 DocumentFile 写入文件
持久化 URI 权限
```

注意：

- 用户可能撤销目录权限
- 目录可能被其他应用修改
- 需要处理文件不存在、权限失效、外部修改冲突

---

## 9.2 应用存储

说明：

```text
仓库保存在 App 私有目录中，其他应用不能直接访问。
卸载 App 后数据会被系统删除。
```

适合用户：

- 只想在 MKM 内使用
- 不关心文件目录
- 更看重简单和安全

UI 文案：

```text
应用存储
保存在 MKM 私有空间中，其他应用不能访问。
卸载 App 后数据会删除，建议开启云同步或定期导出。
```

技术建议：

```text
context.filesDir / repositories/{repositoryId}
```

注意：

- 必须提示卸载会删除数据
- 后续应提供“导出仓库”能力
- 若用户未开启同步，应提醒定期备份

---

## 9.3 V1 默认推荐

V1 推荐默认选中：

```text
设备存储
```

原因：

- 更符合 Markdown 文件管理工具定位
- 更接近 Obsidian Vault 心智
- 用户可以用其他应用访问 Markdown 文件
- 即使 App 出问题，文件仍然可见

但如果用户是新手，也可以选择应用存储，降低目录选择复杂度。

# 11. 本地初始化目录结构建议

V1 创建仓库时，可以初始化以下结构：

```text
MKM-Repository/
├── documents/
│   └── welcome.md
├── todos/
│   └── todos.json
├── assets/
├── .mkm/
│   ├── config.json
│   ├── repository.json
│   └── sync.json
└── README.md
```

说明：

| 路径 | 作用 |
|---|---|
| documents/ | Markdown 文档 |
| todos/ | 本地待办数据或导出数据 |
| assets/ | 图片、附件 |
| .mkm/ | MKM 私有配置 |
| README.md | 仓库说明 |

V1 如果数据库仍以 Room 为主，也建议至少保留 `.mkm/repository.json`，用于识别有效仓库。

# 16. 当前结论

MKM 首次启动 UI 应调整为：

```text
先选择仓库，再决定是否登录。
```

最终确认的产品策略：

```text
1. App 第一屏是仓库初始化，不是登录页
2. App 可以完全不登录使用
3. 账号是可选的
4. 云同步必须登录/注册账号
5. 使用现有仓库支持两种方式：本地目录 + 账号云端恢复
6. 创建仓库时需要设置仓库名称和存储位置
7. 存储位置支持：设备存储、应用存储
8. 设备存储允许其他应用访问
9. 应用存储更私密，但卸载后数据删除
10. 底部提供语言选择入口
```

这套设计让 MKM 更接近真正的 Markdown 知识库产品，而不是一个必须登录的普通云笔记应用。
